You are a senior product engineer at a top-tier enterprise software firm with deep expertise in financial technology, document intelligence, and AI-powered tooling. You are building a production-ready internal tool for Jupiter Asset Management, one of the UK's leading active asset managers. Your standard is high: the UI must look like it belongs in a Bloomberg terminal or a tier-one investment bank's internal portal. No startup aesthetics, no rounded pastel cards, no generic SaaS feel. Every design decision should reinforce that this is a serious financial tool built for professionals.
Build a complete, single-file RAG (Retrieval Augmented Generation) application that allows Jupiter staff to interrogate fund documents in plain English. The output must be a single self-contained HTML file with all JavaScript and CSS inline. No external dependencies except PDF.js and the Anthropic API.
The person using this tool day-to-day is a fund manager, compliance officer, or client relations manager at Jupiter. They are time-poor, detail-oriented, and have zero tolerance for tools that hallucinate, crash, or feel unfinished. Every answer must be grounded in the document. Every failure must be handled gracefully. Every interaction must feel instant and polished.
The person seeing this demo for the first time is an AI transformation lead who needs to understand the business case in under 60 seconds without any technical explanation. The tool must sell itself.
API requirements:

Anthropic API called directly from the browser via fetch
Always include the header anthropic-dangerous-direct-browser-access: true in every single API call without exception
Model: claude-sonnet-4-20250514
Max tokens: 1500

API key gate:

If no API key is stored in sessionStorage, show a clean full-screen key entry screen as the absolute first thing that renders — nothing else should be visible
On entry, validate the key format before storing
Store in sessionStorage only — never in localStorage, never hardcoded

RAG pipeline — this is the technical core, build it properly:

Extract text from PDFs page by page using PDF.js
Chunk extracted text into segments of approximately 420 words with 55 word overlap between chunks to preserve context across boundaries
At query time score all chunks using TF-IDF to find the most semantically relevant sections
Pass only the top 8 ranked chunks to Claude, each labelled with its source page number
Claude must answer strictly from the provided chunks only — the system prompt must explicitly forbid drawing on general knowledge or training data
Every answer must include a citation showing which page and section it came from
If the answer is not present in the chunks Claude must say so explicitly — hallucination is a hard failure mode for this audience
Chunk and prioritise gracefully for large documents — never crash or timeout on a long PDF

Document health check:
Immediately after a PDF loads run a single API call that checks for the following and displays results as a dashboard above the chat before any questions are asked. Show a green tick with a one line summary if present, red cross if absent:

Key risks identified
Regulatory obligations found
Performance data present
Fund manager named
Fees and charges present
Investment objective stated

Pre-load behaviour:
Always show the upload zone on load. Do not attempt to auto-fetch any document. The user should always upload their own PDF manually.
